# speckit-bridge

A skill that converts the output of requirements_designer (`designs/`) into [GitHub spec-kit](https://github.com/github/spec-kit) specification format.

## What This Skill Does

Automatically converts requirement documents (7 files) created by PMs with `/requirements_designer` into a format (`spec.md` + `constitution.md`) that engineers can use with `specify plan → tasks → implement`.

```
/requirements_designer → designs/ (PM deliverables)
        ↓
/speckit-bridge → spec.md + constitution.md (Engineer input)
        ↓
/speckit.plan → plan.md, data-model.md, contracts/
        ↓
/speckit.tasks → tasks.md
        ↓
/speckit.implement → code
```

## Prerequisites

- `designs/` must be generated by `/requirements_designer`
- Quality score of 70 or above (recommended)
- spec-kit CLI must be installed

```bash
# Install spec-kit CLI (first time only)
# ⚠️ The "specify" package on PyPI is a different package. Always use this command.
uv tool install specify-cli --from "git+https://github.com/github/spec-kit.git@v0.4.3"

# Install uv if not already available
curl -LsSf https://astral.sh/uv/install.sh | sh
```

## Conversion Mapping

| designs/ File | Target | Content |
|---|---|---|
| `README.md` | `constitution.md` | Project principles & constraints |
| `README.md` | `spec.md` Success Criteria | Success metrics → SC-001 format |
| `functional_requirements.md` | `spec.md` Functional Requirements | FR-001 → "System MUST" format |
| `user_stories.md` | `spec.md` User Scenarios | US → User Story (P1/P2/P3) |
| `non_functional_requirements.md` | `spec.md` Assumptions | NFR targets as constraints |
| `ubiquitous_language.md` | `spec.md` Key Entities | UL terms → entities |
| `ui_design_brief.md` | Not converted | Managed separately as Figma deliverables |

## Automatic Project Structure Setup

When the bridge is invoked, it automatically checks the project directory structure and suggests organization:

- Displays directory existence status in table format
- Checks `.gitignore` and suggests missing entries (`.claude/`, etc.)
- Suggests adding directory conventions to the project's `CLAUDE.md`

```
my-project/
├── designs/         ← PM deliverables (Git tracked: Yes)
├── .specify/        ← spec-kit config (Git tracked: Yes)
├── specs/           ← Specifications & plans (Git tracked: Yes, feature branch)
├── src/             ← Implementation code (Git tracked: Yes)
├── tests/           ← Tests (Git tracked: Yes)
├── .claude/         ← Claude Code config (Git tracked: No)
├── .gitignore
├── CLAUDE.md
└── package.json
```

All changes are executed only after user confirmation (no files are modified without approval).

## Quality Gate

- Quality score 70+: Convert directly
- Quality score below 70: Show warning, continue at user's discretion
- Quality score not calculated (`-/100`): Recommend running Phase 4A of `/requirements_designer`
